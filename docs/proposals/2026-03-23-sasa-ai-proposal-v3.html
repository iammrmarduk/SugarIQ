— Executive Summary

A Professional Services Engagement

This proposal outlines a ten-session Applied AI Consulting and Development Programme delivered at SASA's Mount Edgecombe facility. This is a professional services engagement under the professional fees budget — not a training initiative.

The focus throughout is on building, integrating, and deploying AI-driven tools and workflows directly into SASA's existing systems and data environment — using SASA's own operational data and building directly on the foundations already established by members of the team. — Approach

How We Work Together

Every session works with SASA's actual data and systems. The deliverables are working tools, tested integrations, documented workflows, and deployable pipelines.

  • We build models using company data, not constructed examples — every exercise uses real SASA datasets
  • Each session produces a tangible deliverable SASA keeps and can use from the following Monday
  • Advanced team members are leveraged as context contributors and peer demonstrators — not held back
  • Sessions accommodate different capability levels simultaneously — some building, others directing and reviewing
  • All co-developed assets (scripts, prompt libraries, specifications) are stored in a shared SASA consulting log accessible to all participants including online attendees
  • Sessions can be paused to consolidate, catch up, or re-focus based on team progress and priorities

— Session Details

What Happens Each Day

Session 1

Foundations Audit & Strategic FramingMapping SASA's data landscape — setting the integration targets that guide all ten sessions

Rather than covering theory, the first session maps SASA's current AI landscape, standardises tooling across the group, and agrees on the specific integration targets every subsequent session will build toward. Existing AI work already completed within SASA is reviewed and used as the benchmark for the whole group.

What we do together
  • Review and contextualise AI work already completed internally — presenting it as the benchmark, not the exception
  • Map SASA's data landscape: R7106 records, LIMS exports, Autolab CSVs, multi-centre historical data, mill reports — identify highest integration potential
  • Configure and standardise tooling: Claude Pro shared account, Copilot licence review, setup for all attendees
  • Establish shared prompt discipline using SASA's own report language and terminology
  • Define programme capstone goals as a group — agreed integration targets (e.g., LIMS exception flagging, automated report commentary)
  • Set up the shared consulting log: folder structure, naming, access for remote attendees
Session deliverable

SASA data & systems map + agreed integration targets document + consulting log established

Session 2

Excel as a Data Engineering SurfaceTurning existing Excel workflows into AI-augmented decision tools

What we do together
  • Rebuild one real SASA Excel report using Claude as live co-pilot: formula logic, data cleaning, conditional exception flagging
  • Introduce structured data principles that make AI tools reason reliably over tabular data
  • Build a reusable Claude prompt library for Excel: summarisation, anomaly detection, trend commentary
  • Apply Microsoft Copilot's Excel integration — compare against Claude direct prompting, establish workflow preference
  • Design an exception-flagging template for one reporting area that a supervisor can operate without technical support
  • (PowerPoint integration also covered — generating slides directly from data. Requires Claude Pro shared account.)
Session deliverable

Rebuilt AI-augmented Excel report + exception-flagging template for one SASA reporting area

Session 3

Working with Large Datasets & RAGMaking 10 years of multi-centre SASA data queryable — without uploading everything to an AI

RAG (Retrieval Augmented Generation) is the pattern that makes large-scale historical analysis practical. This session introduces the concept through hands-on work with SASA's own multi-year, multi-centre data.

What we do together
  • Introduce RAG: how AI answers questions about large data collections without being given the entire dataset
  • Work directly with SASA's multi-year, multi-centre exports — slice, filter, summarise, surface where data structures create friction
  • Build a simple document index: targeted AI queries against the right data subset rather than everything at once
  • Introduce Python: a 5-line AI-generated script reshaping datasets that would take hours in Excel — no prior coding needed
  • Design a data-preparation checklist for SASA's specific formats: LIMS, Autolab, mill reports
Session deliverable

SASA data-preparation checklist + working RAG demonstration against internal SASA documents

Session 4

Python & Node.js as Integration GlueWriting and understanding the code that connects SASA's systems to AI

Managers direct and review the code; the AI writes it. This session builds the literacy to commission AI-generated pipelines confidently.

What we do together
  • Build a real Python data pipeline: LIMS export → transformation → flagged output → formatted report, all AI-generated
  • Introduce Node.js: Python for data, Node.js for API and web-facing connectors to SharePoint and source systems
  • Each participant identifies one repetitive data task for automation — these feed Sessions 7 and 8
IT requirement: Python and Node.js must be pre-installed on attendee machines before this session
Session deliverable

Working LIMS-to-output data pipeline stub + automation candidate list per team area

Session 5

PowerPoint, Reporting & AI-Generated NarrativesEliminating manual report commentary and presentation preparation

What we do together
  • Audit reporting burden: quantify hours per week writing the same interpretive text — the ROI conversation
  • Build a repeatable workflow: structured data → Claude prompt → draft narrative → human review and sign-off
  • Apply to one actual SASA periodic report — produce a complete draft in-session
  • Use Copilot's PowerPoint integration to generate a presentation from a SASA summary — refined through prompt iteration
  • Design a report commentary template encoding SASA's house style, regulatory language, and exception conventions
Session deliverable

AI-assisted SASA report draft + reusable report commentary prompt template

Session 6

Running AI on SASA InfrastructureDeploying AI models inside SASA's environment — no data leaves the building

This session incorporates findings from the IT infrastructure meeting and builds on SASA's data boundary decisions from Session 3.

What we do together
  • Install and configure a local AI model on SASA's server with IT present — participants understand what they are commissioning
  • Walk through Open WebUI: all team members access the AI via browser — no software on individual machines
  • Design SASA's data governance framework: what goes to Claude Pro, what stays on-premise — one-page policy for director sign-off
  • Build a local RAG demonstration: query SASA internal documents without any internet connection
  • Establish the hosting and maintenance model: ownership, upgrade path, SharePoint role

Session 7

LIMS & Autolab IntegrationConnecting AI directly to SASA's laboratory management systems

What we do together
  • Map the LIMS data flow end to end — identify where exceptions are caught late or missed entirely
  • Build an AI-assisted exception detection layer: define business rules with laboratory managers, implement as a Python script or prompt chain
  • Design the notification architecture: exception flagged → channel (email, Teams, SharePoint) → alert content
  • Address Autolab integration: map where AI intercepts instrument data upstream of LIMS (calibration drift, anomaly detection)
  • Produce a written integration specification IT can implement or commission independently
Session deliverable

LIMS exception detection workflow + Autolab integration specification

Session 8

Agentic Workflows & Automation PipelinesAI processes that run without constant human instruction

What we do together
  • Distinguish prompt from agent — multi-step autonomous decision-making using SASA's LIMS workflow as reference
  • Build a multi-step agentic workflow: data arrives → AI checks rules → exception detected → draft alert generated → human reviews → acts
  • Design failure modes and human override: safe failure, correct escalation, audit trail — critical for a regulated industry
  • Each participant maps one workflow for conversion from manual review to agentic exception flagging
Session deliverable

Working agentic exception-flagging workflow + automation pipeline maps per team area

Session 9

Source System Integration & Production DeploymentEmbedding AI at the point of data creation — not after the fact

What we do together
  • Take two or three workflow maps from Session 8 and build functional integration prototypes: real data, real rules, real output channels
  • Build a lightweight API connector (Node.js or Python) allowing SASA's AI layer to query source systems without manual exports
  • Review the Autolab-to-LIMS chain with prototypes in hand — partial live implementation where feasible
  • Produce SASA's AI architecture diagram: which systems talk to which AI layer, what data moves where, what humans approve
Session deliverable

Working integration prototypes + SASA AI architecture diagram

Session 10

Capstone Review, Governance & RoadmapConsolidating what was built — establishing how it is sustained

Morning — showcase
  • Each participant or team presents their best deliverable from the programme (5 minutes each) — directors in attendance
  • Structured review against Session 1 targets: outcome versus ambition, what remains
Afternoon — governance & roadmap
  • Build SASA's AI governance framework: acceptable use, data classification, prompt library ownership, model maintenance
  • Design the ongoing operating model: who maintains what, how new use cases are prioritised
  • 12-month roadmap: three to five integration targets sequenced by value and complexity, with owners and effort estimates
Session deliverable

AI governance framework + 12-month integration roadmap + individual 90-day action plans

— Appendix

AI Terminology Reference

A plain-language glossary of terms used throughout this programme — to keep all participants oriented regardless of technical background.

TermPlain-language definition
AISoftware that understands language, analyses data, and generates outputs without explicit rules for every situation
LLMLarge Language Model — the type of AI powering Claude, ChatGPT, and Copilot
ClaudeAnthropic's AI assistant — the primary tool used in this programme
Claude ProPaid Claude subscription (~$100/month) giving access to the most capable models and higher usage limits
PromptThe instruction or question you give to an AI — how it is phrased significantly affects output quality
Prompt EngineeringThe practice of crafting precise prompts to get reliable, high-quality AI outputs consistently
Context WindowHow much text an AI can "see" at once — larger windows allow working with longer documents and more data
RAGRetrieval Augmented Generation — AI retrieves relevant information from a document store before responding; enables querying large private datasets
TokenThe unit an AI processes — roughly one word or part of a word; pricing and context limits are measured in tokens
CopilotMicrosoft's AI integration in Microsoft 365 (Word, Excel, PowerPoint, Outlook) — SASA already holds licences
APIApplication Programming Interface — a structured way for one system to talk to another; used to connect Claude to Excel, LIMS, or other systems
PythonA programming language suited to data manipulation and automation — all code in this programme is AI-generated; no prior experience required
Node.jsA JavaScript environment for building API integrations and web-facing connectors
VS CodeFree code editor from Microsoft used to review, run, and manage AI-generated scripts
Claude Code CLITerminal-based tool for running Claude from a command line or code environment
WindsurfAn AI-powered code editor where the AI participates in writing and explaining code as you work
OllamaFree, open-source tool for running AI language models locally on a server — no internet or cloud required
Open WebUIBrowser-based interface for accessing a local Ollama model — looks like ChatGPT but runs entirely on SASA's own hardware
Agent / Agentic WorkflowAI set up to complete a multi-step task autonomously — e.g., check LIMS data, detect exception, draft alert, send it — without a human prompting each step
MCPModel Context Protocol — an open standard connecting AI models to external tools and data sources in a controlled, auditable way
n8nOpen-source, self-hostable workflow automation — like Zapier but runs on SASA's own infrastructure at no cost
LIMSLaboratory Information Management System — SASA's laboratory data system; a key integration target for AI exception detection
AutolabSASA's laboratory instrument automation system; feeds data into LIMS; upstream AI integration point for anomaly detection
EmbeddingNumerical representation of text enabling AI to measure similarity between documents — the mechanism behind RAG retrieval
Fine-tuningTraining an AI on specific data — not used here; prompting and RAG achieve the same goals without the cost and complexity
SharePointMicrosoft's document and collaboration platform already in use at SASA — used as a notification and document layer in this programme
